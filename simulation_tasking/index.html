    Step 6: Create Environment Configuration File#

    Run the following command in your terminal: 

    make hatchet-token

    • Copy your client token

    • Paste it into the .env.local.host.hatchet amd the .env.local.hatchet files: 

      HATCHET_CLIENT_TOKEN=your_token_here

    • Save the .env file

    Step 5: Start the workers#

    Check that this step runs. When it does, run the following command:

       make engine

    If you are on a Mac, you may see this error the first time you do this 

    target simulations: failed to solve: image "123456789012.dkr.ecr.us-east-1.amazonaws.com/hatchet/globi:latest": already exists

make: *** [engine] Error 1

    If you encounter this, try re-running the make engine command - this should solve the issue.

    If you want to stop the server, you can run this: 

    hatchet server stop

    Step 7: Run a Test Simulation#

    Now you're ready to run your first simulation using the mock data!

    1. Make sure both Hatchet workers are running

    2. Run the simulation command:

    make cli submit manifest -- --path {PATH_TO_MANIFEST} --grid-run

    1. Monitor the simulation progress:

    2. Open the Hatchet UI in your browser (http://localhost:8080)

    3. Navigate to the "Workflows" or "Runs" section
    4. You should see your simulation job appear

    5. Watch as it gets allocated to a worker and begins processing

    6. Wait for the simulation to complete:

    7. The status will change from "Running" to "Completed" when finished
    8. You can click on the workflow to see detailed logs and outputs

    Once you are done with the simulation runs, you can stop the Docker images by running: 

    make down

    Step 8: View simulation results#

    AFter running the simulation, you should the following in your terminal: 

    versioned_experiment:
  base_experiment:
    experiment: scythe_experiment_simulate_globi_building
    run_name: Cambridge_UK/dryrun/Baseline
    storage_settings:
      BUCKET: globi-bucket
      BUCKET_PREFIX: globi
  version:
    major: 1
    minor: 0
    patch: 0
timestamp: '2026-01-23T13:57:36.233154'

    Copy the run_name so that you can use it for the results get command.

    make cli get experiment  --run-name {YOUR_RUN_NAME_HERE}

    If you have done lots of different simulations, and you want to access specific versions of your runs, then check what version you want to access. It should be in the form x.x.x., e.g., if the version is major.minor.patch.

    You have the ability to also choose your desired output_dir to save to a specific location. Overall, the request becomes: 

    make cli get experiment  --run-name {YOUR_RUN_NAME_HERE} --version {VERSION} --output_dir {YOUR_CHOSEN_OUTPUT_DIR}

    You will see a message telling you where the results have been saved to.

    Troubleshooting#

    Docker Issues#

    • Docker daemon not running: Make sure Docker Desktop is running before executing commands
    • Permission denied: On Linux, ensure your user is in the docker group (see Linux Docker installation step)

    EnergyPlus Issues#

    • EnergyPlus not found: Verify EnergyPlus is in your system PATH

    Hatchet Issues#

    • Workers not appearing: Check that both worker terminals are running without errors
    • Connection refused: Ensure the Hatchet server is running and accessible at http://localhost:8080
    • Token errors: Verify your HATCHET_CLIENT_TOKEN in the .env file is correct

    Python/uv Issues#

    • Module not found: Run uv sync --all-extras --all-groups again to ensure all dependencies are installed
    • Python version error: Verify you have Python 3.12 or higher with python --version

    Quick Reference#

    Essential Commands#

    # Start simulation worker
make hatchet-simulation-worker

# Start fanouts worker (in separate terminal)
make hatchet-fanouts-worker

# Run a simulation
hatchet worker dev --run-cmd "uv run --env-file .env globi submit manifest --path data/partners/LOCATION/manifest.yml --grid-run"

# Check Hatchet UI
open http://localhost:8080
